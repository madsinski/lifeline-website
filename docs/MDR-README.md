# Lifeline — MDR-relevant infrastructure overview

> Audience: notifying body auditors, internal regulatory leads,
> QMS reviewers, and any future contributor who needs to find
> the design-history-file evidence quickly.

This document maps the technical artifacts that support an EU MDR
2017/745 (and IS implementation) submission for the Lifeline app
and platform. It is **not** the technical file itself — that lives
in the regulatory drive — but it tells you where every piece of
machine-generated evidence is stored, so the human-written sections
of the technical file can cite specific tables, buckets, and
filesystem paths.

## Two repositories

| Repo | Role | CHANGELOG |
|------|------|-----------|
| `fhir-health-dashboard` | React Native mobile app (Expo) | `/CHANGELOG.md` |
| `lifeline-website` | Next.js admin + public site + Supabase schema | `/CHANGELOG.md` |

Both `CHANGELOG.md` files follow [Keep a Changelog](https://keepachangelog.com/)
and append-only. Each release block has a **Risk-relevant** section
listing changes that affect safety, clinical accuracy, regulatory
posture, data integrity, or audit trail. Those entries cross-
reference a `risk_register` row by its short id (`R-YYYY-NNN`).

## Versioning policy

Semver (`MAJOR.MINOR.PATCH`):

- **MAJOR** — incompatible data changes, removed clinical capability,
  change of intended use.
- **MINOR** — new clinical capability or material UX change.
- **PATCH** — bug fixes + non-clinical polish.

The mobile app version lives in `app.config.js` → `expo.version`.
EAS auto-increments `buildNumber` (iOS) and `versionCode` (Android)
per channel. The website doesn't ship binaries — its "version" is
the Vercel deployment id or a release tag pushed by hand.

## Per-release artifacts

For every shipped build there's one row in `app_releases` and one
SBOM blob in the `app-releases-sbom` Supabase storage bucket.

| Column | What it captures |
|---|---|
| `repo` | `app` or `website` |
| `version` | semver |
| `build_number` | iOS CFBundleVersion / Android versionCode / Vercel deploy id |
| `platform` | `ios` / `android` / `all` / `web` |
| `channel` | `development` / `preview` / `production` |
| `git_sha` | full 40-char commit hash — the immutable link back to source |
| `git_tag` | optional semver tag (e.g. `v0.1.0`) |
| `git_branch` | branch the release was cut from |
| `release_notes` | markdown copy of the CHANGELOG entry for this release |
| `risk_assessment` | summary of risk-relevant changes + register short_ids touched |
| `sbom_storage_path` | path in `app-releases-sbom` (JSON, generated by `npm ls --all --json`) |
| `sbom_sha256` | integrity hash so the file can't be silently swapped |
| `build_artifact_url` | EAS build URL / Vercel deployment URL |
| `released_at`, `released_by`, `released_by_email` | who shipped it and when |

Unique constraint: `(repo, version, build_number, platform, channel)`.
The release recorder is idempotent — running it twice with the same
arguments returns HTTP 409 without creating a duplicate.

### Recording a release

```bash
# From the RN app repo:
bash scripts/release.sh \
  --channel preview \
  --platform android \
  --build-number 12 \
  --build-url 'https://expo.dev/...'
```

The script prompts for risk assessment + release notes, generates
the SBOM, and posts to `/api/admin/releases`. View the result at
`https://www.lifelinehealth.is/admin/releases`.

## Risk register (ISO 14971)

`risk_register` rows are the working version of the formal risk
management file. Each row:

- Has an auto-assigned short id (`R-YYYY-NNN`) for cross-referencing.
- Captures initial severity + likelihood + risk score.
- Records the mitigation (what we do or plan to do).
- Records the residual severity / likelihood / score after mitigation.
- Records the detection mechanism (how we'd know if the risk materializes).
- Has a status workflow: `open` → `mitigated` / `closed` / `accepted`.

Read/write surface: `/admin/releases` → Risk register tab.
DB write surface: `public.risk_register` with RLS via
`is_active_staff()` (read) and `is_admin_staff()` (write).

## Consent + signature audit trail

| Table | Purpose | PDF storage bucket |
|-------|---------|--------------------|
| `platform_agreement_acceptances` | TOS / DPA / health-assessment consent (every user) | `platform-acceptance-pdfs` |
| `staff_agreement_acceptances` | NDA / acceptable-use for team members | `staff-acceptance-pdfs` |
| `client_consents` | Per-feature health consents (e.g. body-comp import) | `client-consent-pdfs` |
| `beta_nda_acceptances` | Beta tester agreement (this beta program only) | `beta-nda-pdfs` |

All four tables share the same audit-grade shape: who signed, what
text version (with sha256 of canonical text), IP, user-agent,
timestamp, and a generated PDF certificate. The PDF is a two-page
A4 with page 1 = audit metadata and page 2 = full document text —
generated server-side via `@react-pdf/renderer` so the user can't
modify what they're "signing".

## Health data audit log

`health_audit_log` is written by triggers on every health-data
table (body comp, blood markers, programs, etc.) plus manual
`log_health_access()` calls from server routes that SELECT health
data. Every row captures actor id, actor email, actor role, action
(`SELECT|INSERT|UPDATE|DELETE`), table name, row id, and metadata.

## Error mirror — `app_errors`

A single triage table for errors from every runtime:

- **mobile** — posted from RN app via `src/services/errorReporter.ts`
  (`POST /api/errors`). Pre-login errors are queued to AsyncStorage
  and flushed after sign-in.
- **server** — posted from Next.js API routes via
  `src/lib/error-reporter.ts` (`captureException` / `captureMessage`)
  using the service role.
- **browser** — posted from the admin / public website via
  `src/app/components/ErrorHandlerClient.tsx`, which installs
  `window.onerror` + `unhandledrejection` handlers
  (`POST /api/errors/capture`).

Sensitive fields (kennitalas, emails inside stack traces) are
redacted server-side before storage. UA + app version are stamped
on each row so an error can be correlated back to a specific
`app_releases` row.

Admin view: `/admin/errors`. Email digest: weekly cron writes to
the same address as security advisories.

## Versioned content

These data sources have explicit `*_version` keys so an auditor
can reproduce exactly what a user saw at a given moment:

- **Legal docs** — `*_VERSION` constants in `src/lib/*-content.ts`
  (TOS, DPA, employee TOS, health-assessment consent, beta NDA).
  Server-rendered every request, hashed at acceptance time.
- **Sales terms** (Söluskilmálar) — versioned page metadata at
  `/soluskilmalar` (IS) and `/sales-terms` (EN).

> **Not yet versioned** — see open items below.

## SECURITY DEFINER helpers

These functions encapsulate cross-table identity lookups that
RLS alone can't safely express. Defined in
`migration-staff-helpers-email-match.sql` and
`migration-get-my-staff-profile.sql`:

- `is_active_staff()` — current auth user is on the staff roster.
- `is_admin_staff()` — current auth user has `staff.role = 'admin'`.
- `is_active_lawyer()` / `is_active_clinician()` /
  `is_active_psychologist()` / `is_active_coach_or_clinician()`.
- `get_my_staff_profile()` — returns the current user's
  `role + permissions + active + email` in one call (used by the
  React `useStaffGuard` hook so the admin UI works for invited
  admins whose `staff.id != auth.users.id`).

## Encryption at rest

`messages.content` and PII columns on `clients` are encrypted via
pgcrypto + Supabase Vault. App-facing code reads through
`messages_decrypted` / `clients_decrypted` views with `INSTEAD OF`
triggers handling writes. No plaintext columns exist. See
`src/lib/security-posture.ts` for the audit-ready statement (bump
`SECURITY_POSTURE_VERSION` on any substantive edit).

## Open items / known gaps

These are tracked outside this file (commonly in `risk_register`
with status `open`):

- iOS Apple HealthKit reader is a stub (`src/services/healthKit.ts`).
  Wearable pairing flow works in the app; reads return empty until
  `@kingstinct/react-native-healthkit` is wired. Beta testers are
  notified up front.
- Server route error capture is opt-in (each handler must call
  `captureException` in its catch block) — there is no
  global wrapper yet.
- Clinical-content versioning (program definitions, marker
  reference ranges, AI prompts) follows the legal-doc pattern in
  some places but isn't fully versioned everywhere.
